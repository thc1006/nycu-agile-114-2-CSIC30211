# CampusEats Acceptance Criteria & Example Mapping

**文件版本：** v1.0  
**負責角色：** PM / Scrum Master  
**適用範圍：** CampusEats MVP  
**對應文件：** `backlog.md`  
**狀態：** Draft  

---

## 1. 文件目的

本文件用來補強 CampusEats MVP 中核心 User Stories 的 Acceptance Criteria 與 Example Mapping。

目的如下：

1. 讓 PM / Scrum Master 與 RD 對需求有共同理解
2. 在 Sprint Planning 前釐清模糊規則
3. 作為 Sprint Review 驗收依據
4. 滿足期末專案中至少 5 個 Story 需撰寫 AC 的要求
5. 協助後續測試與 Demo 腳本設計

---

## 2. Example Mapping 格式

每個 Story 會包含以下內容：

| 區塊 | 說明 |
|---|---|
| Story | User Story 本身 |
| Rules | 此 Story 必須遵守的業務規則 |
| Examples | 用具體情境說明系統行為 |
| Questions | 尚未釐清、需和團隊討論的問題 |
| Acceptance Criteria | 可作為驗收依據的 Given-When-Then |

# AG-002 發布訂單

## Story

身為訂餐者，  
我想要發布餐點需求訂單，  
以便讓帶餐者知道我需要什麼餐點、送到哪裡，以及我願意支付多少帶餐費。

---

## Rules

| Rule ID | Rule |
|---|---|
| R-002-01 | 使用者必須登入後才能發布訂單 |
| R-002-02 | 發布訂單時必須填寫餐廳、餐點內容、取餐地點、期望時間與帶餐費 |
| R-002-03 | 訂單建立成功後，初始狀態必須為「待接單」 |
| R-002-04 | 帶餐費必須為有效數字，且不可小於 0 |
| R-002-05 | 訂單建立成功後，應出現在待接訂單列表 |
| R-002-06 | 未完成必填欄位時，不得建立訂單 |
| R-002-07 | 訂餐者應能在送出前確認自己填寫的訂單資訊 |

---

## Examples

| Example ID | 情境 | 輸入 | 預期結果 |
|---|---|---|---|
| EX-002-01 | 成功發布訂單 | 餐廳：二餐、餐點：雞腿便當、地點：資工系館、時間：12:30、帶餐費：20 | 建立訂單，狀態為待接單 |
| EX-002-02 | 缺少餐點內容 | 餐點欄位空白 | 系統提示餐點內容必填，不建立訂單 |
| EX-002-03 | 缺少取餐地點 | 取餐地點空白 | 系統提示取餐地點必填，不建立訂單 |
| EX-002-04 | 帶餐費為負數 | 帶餐費：-10 | 系統提示帶餐費格式錯誤，不建立訂單 |
| EX-002-05 | 帶餐費為文字 | 帶餐費：abc | 系統提示帶餐費格式錯誤，不建立訂單 |
| EX-002-06 | 訂單建立後列表可見 | 成功建立一筆待接單 | 帶餐者可在待接列表看到該訂單 |
| EX-002-07 | 未登入發布訂單 | 使用者未登入但嘗試進入發單頁 | 系統導向登入頁或提示需登入 |

---

## Questions

| Question ID | 問題 | 影響 |
|---|---|---|
| Q-002-01 | 期望時間是否需要限制只能選未來時間？ | 影響表單驗證 |
| Q-002-02 | 帶餐費是否允許 0 元？ | 影響接單誘因與表單規則 |
| Q-002-03 | 餐廳名稱是自由輸入還是固定清單？ | 影響 UI 與資料庫設計 |
| Q-002-04 | 餐點內容是否要限制字數？ | 影響資料儲存與 UI 顯示 |
| Q-002-05 | 是否需要訂單確認頁？ | 影響發單流程長度 |

---

## Acceptance Criteria

```gherkin
Scenario: 訂餐者成功發布訂單
Given 使用者已登入
When 使用者填寫餐廳、餐點內容、取餐地點、期望時間與帶餐費
And 使用者送出訂單
Then 系統建立一筆新訂單
And 訂單狀態為「待接單」
And 訂單會出現在待接訂單列表

Scenario: 訂單必填欄位未填
Given 使用者已登入
When 使用者未填寫任一必填欄位
And 使用者送出訂單
Then 系統提示欄位錯誤
And 系統不建立訂單

Scenario: 帶餐費格式錯誤
Given 使用者已登入
When 使用者輸入非數字或小於 0 的帶餐費
And 使用者送出訂單
Then 系統提示帶餐費格式錯誤
And 系統不建立訂單

Scenario: 未登入使用者不可發布訂單
Given 使用者尚未登入
When 使用者嘗試進入發布訂單頁
Then 系統要求使用者先登入
```

# AG-003 瀏覽待接訂單列表

## Story

身為帶餐者，  
我想要瀏覽目前可接的訂單列表，  
以便找到適合我順路帶餐的訂單。

---

## Rules

| Rule ID | Rule |
|---|---|
| R-003-01 | 使用者必須登入後才能查看訂單列表 |
| R-003-02 | 待接訂單列表只顯示狀態為「待接單」的訂單 |
| R-003-03 | 已接單、購買中、已送達、已完成、已取消的訂單不應出現在待接列表 |
| R-003-04 | 每筆訂單至少顯示餐廳、餐點摘要、取餐地點、期望時間與帶餐費 |
| R-003-05 | 訂單列表預設以期望時間由近到遠排序 |
| R-003-06 | 若沒有待接訂單，系統需顯示空狀態提示 |

---

## Examples

| Example ID | 情境 | 資料狀態 | 預期結果 |
|---|---|---|---|
| EX-003-01 | 有三筆待接訂單 | 三筆訂單狀態皆為待接單 | 列表顯示三筆訂單 |
| EX-003-02 | 有一筆已接單 | 訂單狀態為已接單 | 不顯示在待接列表 |
| EX-003-03 | 有一筆已完成訂單 | 訂單狀態為已完成 | 不顯示在待接列表 |
| EX-003-04 | 沒有任何待接訂單 | 無待接單資料 | 顯示「目前沒有可接訂單」 |
| EX-003-05 | 多筆訂單時間不同 | 12:10、12:30、13:00 | 依 12:10、12:30、13:00 排序 |
| EX-003-06 | 訂單資訊不完整 | 缺少餐廳或地點 | 該訂單不可被發布或列表需顯示錯誤資料 |

---

## Questions

| Question ID | 問題 | 影響 |
|---|---|---|
| Q-003-01 | 訂單列表是否需要顯示完整餐點內容？ | 影響隱私與列表可讀性 |
| Q-003-02 | 是否需要篩選功能？ | 影響 Sprint 1 範圍 |
| Q-003-03 | 訂單排序是依期望時間還是建立時間？ | 影響列表邏輯 |
| Q-003-04 | 訂餐者是否可以看到自己發布的訂單也在列表中？ | 影響角色與權限 |
| Q-003-05 | 帶餐者是否能看到報酬為 0 的訂單？ | 影響接單誘因 |

---

## Acceptance Criteria

```gherkin
Scenario: 顯示待接訂單
Given 使用者已登入
When 使用者進入待接訂單列表頁
Then 系統顯示所有狀態為「待接單」的訂單
And 每筆訂單顯示餐廳、餐點摘要、取餐地點、期望時間與帶餐費

Scenario: 不顯示已接單訂單
Given 系統中存在一筆狀態為「已接單」的訂單
When 使用者查看待接訂單列表
Then 該訂單不會出現在列表中

Scenario: 訂單列表排序
Given 系統中存在多筆待接訂單
When 使用者進入待接訂單列表
Then 系統依期望時間由近到遠排序

Scenario: 沒有待接訂單
Given 系統中沒有任何待接訂單
When 使用者進入待接訂單列表
Then 系統顯示「目前沒有可接訂單」
```

# AG-004 接單

## Story

身為帶餐者，  
我想要接下某一筆待接訂單，  
以便確認我會負責這筆帶餐需求。

---

## Rules

| Rule ID | Rule |
|---|---|
| R-004-01 | 只有登入使用者可以接單 |
| R-004-02 | 只有狀態為「待接單」的訂單可以被接單 |
| R-004-03 | 接單成功後，訂單狀態變為「已接單」 |
| R-004-04 | 接單成功後，系統需記錄接單者 |
| R-004-05 | 同一筆訂單只能被一位帶餐者接單 |
| R-004-06 | 訂餐者不能接自己的訂單 |
| R-004-07 | 接單後，該訂單不應再出現在其他帶餐者的待接列表 |
| R-004-08 | 接單者可以查看完整訂單資訊 |

---

## Examples

| Example ID | 情境 | 初始狀態 | 操作 | 預期結果 |
|---|---|---|---|---|
| EX-004-01 | 成功接單 | 訂單為待接單 | 帶餐者點擊接單 | 狀態變為已接單 |
| EX-004-02 | 重複接單 | 訂單已被 A 接單 | B 嘗試接單 | 系統提示已被接單 |
| EX-004-03 | 訂餐者接自己的單 | 訂單由小美建立 | 小美點擊接單 | 系統不允許 |
| EX-004-04 | 未登入接單 | 使用者未登入 | 點擊接單 | 系統要求登入 |
| EX-004-05 | 接單後查看詳情 | 接單成功 | 進入詳情頁 | 顯示完整餐點資訊 |
| EX-004-06 | 已取消訂單被接單 | 訂單狀態已取消 | 帶餐者點擊接單 | 系統不允許 |

---

## Questions

| Question ID | 問題 | 影響 |
|---|---|---|
| Q-004-01 | 是否需要接單確認頁？ | 避免誤接單 |
| Q-004-02 | 帶餐者是否可同時接多筆訂單？ | 影響狀態與資料結構 |
| Q-004-03 | 訂餐者是否要收到接單通知？ | 影響通知功能 |
| Q-004-04 | 接單後是否能取消接單？ | 影響例外流程 |
| Q-004-05 | 是否需要顯示接單者名稱給訂餐者？ | 影響信任與隱私 |

---

## Acceptance Criteria

```gherkin
Scenario: 帶餐者成功接單
Given 使用者已登入
And 訂單狀態為「待接單」
When 使用者點擊「接單」
Then 訂單狀態變為「已接單」
And 系統記錄該使用者為接單者

Scenario: 同一筆訂單不可重複接單
Given 訂單狀態已變為「已接單」
When 其他使用者嘗試接單
Then 系統提示該訂單已被接
And 系統不允許重複接單

Scenario: 訂餐者不可接自己的訂單
Given 使用者是該訂單的建立者
When 使用者嘗試接單
Then 系統不允許接單
And 顯示錯誤提示

Scenario: 接單後可查看完整訂單
Given 使用者成功接下一筆訂單
When 使用者進入訂單詳情頁
Then 系統顯示完整餐點內容、取餐地點、期望時間與帶餐費
```

# AG-005 更新訂單狀態

## Story

身為帶餐者，  
我想要更新訂單狀態，  
以便讓訂餐者知道目前餐點購買與送達進度。

---

## Rules

| Rule ID | Rule |
|---|---|
| R-005-01 | 只有接單者可以更新訂單狀態 |
| R-005-02 | 訂單狀態必須依照固定順序變更 |
| R-005-03 | 狀態流程為：待接單 → 已接單 → 購買中 → 已送達 → 已完成 |
| R-005-04 | 「已接單」只能更新為「購買中」 |
| R-005-05 | 「購買中」只能更新為「已送達」 |
| R-005-06 | 訂餐者可以查看最新狀態 |
| R-005-07 | 已完成或已取消的訂單不可再更新狀態 |

---

## Examples

| Example ID | 情境 | 初始狀態 | 操作 | 預期結果 |
|---|---|---|---|---|
| EX-005-01 | 開始購買 | 已接單 | 點擊開始購買 | 狀態變為購買中 |
| EX-005-02 | 標記送達 | 購買中 | 點擊已送達 | 狀態變為已送達 |
| EX-005-03 | 狀態跳躍 | 已接單 | 直接改成已送達 | 系統不允許 |
| EX-005-04 | 非接單者更新 | 已接單 | 非接單者點擊更新 | 系統不允許 |
| EX-005-05 | 訂餐者查看狀態 | 狀態為購買中 | 訂餐者查看詳情 | 顯示購買中 |
| EX-005-06 | 已完成訂單更新 | 已完成 | 接單者嘗試更新 | 系統不允許 |

---

## Questions

| Question ID | 問題 | 影響 |
|---|---|---|
| Q-005-01 | 狀態是否需要「送達途中」？ | 影響流程複雜度 |
| Q-005-02 | 狀態變更是否需要時間紀錄？ | 影響資料欄位 |
| Q-005-03 | 若帶餐者忘記更新狀態怎麼辦？ | 影響提醒機制 |
| Q-005-04 | 訂餐者是否能手動提醒帶餐者更新？ | 影響通知功能 |
| Q-005-05 | 是否要記錄狀態變更歷史？ | 影響後續追蹤與糾紛處理 |

---

## Acceptance Criteria

```gherkin
Scenario: 更新為購買中
Given 使用者是該訂單的接單者
And 訂單狀態為「已接單」
When 使用者點擊「開始購買」
Then 訂單狀態變為「購買中」

Scenario: 更新為已送達
Given 使用者是該訂單的接單者
And 訂單狀態為「購買中」
When 使用者點擊「已送達」
Then 訂單狀態變為「已送達」

Scenario: 非接單者不可更新狀態
Given 使用者不是該訂單的接單者
When 使用者嘗試更新訂單狀態
Then 系統不允許更新

Scenario: 狀態不可跳躍
Given 訂單狀態為「已接單」
When 使用者嘗試直接將訂單更新為「已送達」
Then 系統不允許此操作

Scenario: 訂餐者可查看最新狀態
Given 帶餐者已更新訂單狀態
When 訂餐者查看訂單詳情
Then 訂餐者可以看到最新訂單狀態
```

# AG-006 確認收餐

## Story

身為訂餐者，

# AG-007 雙向評價

## Story

身為訂餐者與帶餐者，  
我想要在交易完成後互相評價，  
以便建立平台的基本信任機制。

---

## Rules

| Rule ID | Rule |
|---|---|
| R-007-01 | 只有已完成訂單可以評價 |
| R-007-02 | 訂餐者只能評價該筆訂單的帶餐者 |
| R-007-03 | 帶餐者只能評價該筆訂單的訂餐者 |
| R-007-04 | 評價星等必須介於 1 到 5 |
| R-007-05 | 每位使用者對同一筆訂單只能評價一次 |
| R-007-06 | 評價送出後不可修改 |

## Acceptance Criteria

```gherkin
Scenario: 訂餐者評價帶餐者
Given 使用者是該訂單的訂餐者
And 訂單狀態為「已完成」
When 使用者給帶餐者 1 到 5 星評分並送出
Then 系統儲存該評價

Scenario: 帶餐者評價訂餐者
Given 使用者是該訂單的帶餐者
And 訂單狀態為「已完成」
When 使用者給訂餐者 1 到 5 星評分並送出
Then 系統儲存該評價

Scenario: 未完成訂單不可評價
Given 訂單狀態不是「已完成」
When 使用者嘗試評價
Then 系統不允許評價

Scenario: 評價星等超出範圍
Given 使用者正在評價
When 使用者輸入小於 1 或大於 5 的星等
Then 系統提示評分必須介於 1 到 5 之間

Scenario: 使用者不可重複評價同一筆訂單
Given 使用者已經對該訂單送出評價
When 使用者再次嘗試評價
Then 系統不允許再次送出評價
```
