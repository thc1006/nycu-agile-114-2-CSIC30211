# CampusEats 校園美食媒合平台 MVP Impact Map

## 1. Impact Mapping 目的

本文件用來說明 CampusEats MVP 的產品目標、主要使用者、期望造成的行為改變，以及為了達成這些影響所需交付的功能。

本階段 Impact Map 聚焦於期末專案 MVP，不涵蓋完整商業化平台、平台金流、GPS 追蹤、完整管理後台、多校區支援或 AI 推薦等後續功能。

---

## 2. Why：為什麼要做？

### 產品目標

在期末展示前，完成一個可操作的校園帶餐媒合 MVP，讓學生能完成一筆從發單、接單、狀態更新、確認收餐到評價的完整交易流程。

### 目標描述

CampusEats 希望解決校園中「有人需要幫忙買餐」與「有人願意順路帶餐」之間缺乏有效媒合機制的問題。

MVP 的重點不是打造完整外送平台，而是先驗證：

1. 訂餐者是否能清楚發布帶餐需求
2. 帶餐者是否能理解訂單資訊並接單
3. 雙方是否能透過狀態更新掌握交易進度
4. 交易完成後是否能用評價建立基本信任
5. 這個流程是否足以支撐後續功能擴充

---

## 3. Success Criteria：成功標準

本專案為期末專案，因此成功標準以可展示、可操作、可驗收為主，而非長期營運數據。

| 成功指標 | 驗收方式 | 目標 |
|---|---|---|
| 核心流程完成度 | Demo 操作 | 可完整跑完「登入 → 發單 → 接單 → 狀態更新 → 確認收餐 → 評價」 |
| Story 完成度 | Sprint Review | P0 stories 符合 AC 並可展示 |
| 使用者理解度 | 測試者操作 / Review 回饋 | 測試者能理解如何發單、接單與完成交易 |
| 狀態可追蹤性 | Demo 操作 | 訂餐者可看到訂單狀態變化 |
| 基本信任機制 | Demo 操作 | 完成交易後雙方可留下星等評價 |
| 敏捷流程證據 | 文件與影片 | 具備 Sprint Planning、Sprint Review、Sprint Retrospective 紀錄 |

## 4. Who：誰會影響目標？

MVP 階段主要聚焦兩類核心使用者。

| Actor | 說明 | 是否為 MVP 核心 |
|---|---|---|
| 訂餐者 | 需要他人協助購買餐點的學生 | 是 |
| 帶餐者 | 本來就要外出用餐，願意順便幫忙帶餐的學生 | 是 |
| 管理員 | 處理異常訂單、使用者管理與平台維護 | 否，列為後續功能 |
| 校方 / 餐廳 | 影響平台推廣、場域使用與餐廳資料正確性 | 否，列為外部利害關係人 |

## 6. Actor 1：訂餐者

### 6.1 目標行為

訂餐者能在平台上清楚表達自己的餐點需求，並在交易過程中掌握訂單狀態，最後確認收餐與給予評價。

### 6.2 Impact 與 Deliverable

| Impact：希望造成的行為改變 | Deliverable：對應功能 | 優先級 | 對應 Story |
|---|---|---|---|
| 訂餐者能使用帳號進入平台 | Email 註冊 / 登入 / 登出 | P0 | AG-001 |
| 訂餐者能發布清楚的餐點需求 | 發布訂單表單 | P0 | AG-002 |
| 訂餐者能知道訂單是否已被接單 | 訂單狀態顯示 | P0 | AG-004 / AG-005 |
| 訂餐者能掌握購買與送達進度 | 狀態追蹤頁 | P0 | AG-005 |
| 訂餐者能確認收到餐點 | 確認收餐按鈕 | P0 | AG-006 |
| 訂餐者能評價帶餐者 | 星等評價 | P1 | AG-007 |
| 訂餐者能取消尚未被接的訂單 | 取消訂單 | P1 | AG-008 |

### 6.3 主要假設

| 假設 | 風險 | 驗證方式 |
|---|---|---|
| 訂餐者願意填寫餐點、地點、時間與報酬 | 表單太麻煩導致不想使用 | Sprint Review 請測試者試填一筆訂單 |
| 訂餐者看得懂訂單狀態 | 狀態命名不清楚 | Demo 時觀察測試者是否理解 |
| 訂餐者願意在收餐後確認完成 | 忘記確認，流程卡住 | 加入明顯的確認收餐按鈕 |
| 訂餐者願意留下評價 | 評價流程太麻煩 | MVP 先做簡單 1–5 星評分 |

## 7. Actor 2：帶餐者

### 7.1 目標行為

帶餐者能快速瀏覽可接訂單，判斷是否順路與報酬是否合理，接單後能更新訂單狀態，並完成交付。

### 7.2 Impact 與 Deliverable

| Impact：希望造成的行為改變 | Deliverable：對應功能 | 優先級 | 對應 Story |
|---|---|---|---|
| 帶餐者能使用帳號進入平台 | Email 註冊 / 登入 / 登出 | P0 | AG-001 |
| 帶餐者能看到目前可接訂單 | 待接訂單列表 | P0 | AG-003 |
| 帶餐者能判斷訂單是否適合接 | 顯示餐廳、取餐地點、期望時間、帶餐費 | P0 | AG-003 |
| 帶餐者能接下某筆訂單 | 接單功能 | P0 | AG-004 |
| 帶餐者能避免接到已被接走的訂單 | 接單狀態鎖定 | P0 | AG-004 |
| 帶餐者能回報購買進度 | 更新為「購買中」 | P0 | AG-005 |
| 帶餐者能回報已送達 | 更新為「已送達」 | P0 | AG-005 |
| 帶餐者能評價訂餐者 | 星等評價 | P1 | AG-007 |

### 7.3 主要假設

| 假設 | 風險 | 驗證方式 |
|---|---|---|
| 帶餐者願意瀏覽訂單列表 | 訂單資訊不足，無法判斷是否接單 | Review 時請測試者說明是否知道該不該接 |
| 帶餐者願意手動更新狀態 | 忘記更新，訂餐者不知道進度 | 將狀態更新設計為主要操作按鈕 |
| 帶餐者只會接自己能完成的訂單 | 亂接單造成交易失敗 | 接單前顯示確認資訊 |
| 帶餐者願意完成評價 | 評價誘因不足 | MVP 先做簡單星等，降低操作成本 |

## 8. MVP Deliverables

MVP 階段只保留能支撐完整交易閉環的功能。

| 編號 | Deliverable | 說明 | 優先級 |
|---|---|---|---|
| D1 | 帳號註冊 / 登入 / 登出 | 使用者能進入平台並保有基本身份 | P0 |
| D2 | 發布訂單 | 訂餐者能建立帶餐需求 | P0 |
| D3 | 待接訂單列表 | 帶餐者能查看可接訂單 | P0 |
| D4 | 接單 | 帶餐者能接下訂單，訂單狀態改變 | P0 |
| D5 | 訂單狀態更新 | 帶餐者能更新「購買中」「已送達」 | P0 |
| D6 | 確認收餐 | 訂餐者能完成交易 | P0 |

| D8 | 取消訂單 | 訂餐者可取消尚未被接的訂單 | P1 |
| D9 | Email / 站內通知 | 狀態變更時通知使用者 | P2 |

## 9. Not Now：暫不處理項目

以下項目雖然可能對完整產品有價值，但不列入本次 MVP Impact Map 的主要交付範圍。

| 項目 | 暫不處理原因 |
|---|---|
| 平台金流 | 法規與實作複雜度高，非 MVP 核心 |
| 即時 GPS 追蹤 | 涉及隱私與技術成本，先用手動狀態更新 |
| 管理後台 | 對展示完整交易流程不是必要 |
| 申訴流程 | 可在未來補充，MVP 先處理正常交易流程 |
| 多人合併訂單 | 流程複雜，容易拖慢開發 |
| 餐廳資料庫管理 | 初期可先用文字輸入或固定選單 |
| 推薦碼 / 成長機制 | 屬於營運成長功能，非核心流程 |
| 帶餐者徽章 / 排名 | 屬於激勵機制，非 MVP 必要 |
| 多校區支援 | 先聚焦單一校園情境 |
| AI 推薦 | 非本次期末主要學習目標 |

## 11. Impact Mapping 結論

本 MVP 的 Impact Map 顯示，CampusEats 第一階段應聚焦於完成「校園帶餐交易閉環」，而非一次建立完整平台。

因此本次期末專案的開發優先順序為：

1. 先讓訂餐者能發布清楚訂單
2. 再讓帶餐者能瀏覽並接單
3. 接著讓雙方能追蹤訂單狀態
4. 最後讓訂餐者確認收餐並完成評價

只要這條流程能在 Sprint Review 中穩定展示，就代表本 MVP 已能支撐期末報告的核心目標。
