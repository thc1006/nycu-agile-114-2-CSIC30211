# CampusEats Story Map + Sprint Slicing

## 1. 文件目的

本文件用來將 CampusEats MVP 的使用者流程拆解為 User Story，並依照 Sprint 進行切分。

本文件的用途包含：

1. 對齊 PM / Scrum Master / RD 對產品流程的理解
2. 作為 Sprint Planning 選取 Sprint Backlog 的依據
3. 確認每個 Sprint 都能產出可展示的產品增量
4. 支援期末報告中的 Story Mapping 與 Sprint 規劃說明

---

## 2. Story Mapping 原則

本 Story Map 依據以下原則整理：

1. 以端到端使用流程為主，不只依角色分功能。
2. 優先完成「發單 → 接單 → 狀態更新 → 確認收餐 → 評價」的完整交易閉環。
3. 每個 Sprint 都要有可展示成果。
4. P0 Story 優先支撐 MVP Demo。
5. P1 / P2 Story 視時間與 Sprint 狀況調整，不影響核心流程完成。

---

## 3. MVP 主流程

CampusEats MVP 的主流程如下：

> 登入 → 發布訂單 → 瀏覽訂單 → 接單 → 更新狀態 → 確認收餐 → 評價

此流程對應兩種主要角色：

| 角色 | 主要行為 |
|---|---|
| 訂餐者 | 登入、發布訂單、查看狀態、確認收餐、評價帶餐者 |
| 帶餐者 | 登入、瀏覽訂單、接單、更新狀態、評價訂餐者 |

## 8. Sprint 1：登入與發單基礎流程

### Sprint Goal

讓使用者能登入平台，訂餐者能發布訂單，帶餐者能看到待接訂單列表。

### Sprint 1 Story

| ID | Story | Priority | Activity | 說明 |
|---|---|---|---|---|
| AG-001 | Email 註冊與登入 | P0 | A1 帳號進入 | 使用者可註冊、登入、登出 |
| AG-002 | 發布訂單 | P0 | A2 發布訂單 | 訂餐者可建立餐點需求 |
| AG-003 | 瀏覽待接訂單列表 | P0 | A3 瀏覽訂單 | 帶餐者可看到目前待接訂單 |

### Sprint 1 Demo Flow

1. 使用者註冊帳號
2. 使用者登入平台
3. 訂餐者建立一筆訂單
4. 系統顯示訂單已建立
5. 切換到帶餐者視角
6. 帶餐者看到待接訂單列表

### Sprint 1 驗收重點

| Story | 驗收重點 |
|---|---|
| AG-001 | 新使用者可註冊；已註冊使用者可登入；登出後不可直接進入平台 |
| AG-002 | 必填欄位完整才能送出；送出後訂單進入待接單列表 |
| AG-003 | 訂單列表只顯示待接單；列表顯示餐廳、餐點摘要、地點、時間、帶餐費 |

### Sprint 1 Review 可收集的回饋

| 問題 | 對象 |
|---|---|
| 發布訂單欄位是否清楚？ | 訂餐者 |
| 帶餐者是否能判斷這筆訂單是否值得接？ | 帶餐者 |
| 訂單列表顯示的資訊是否足夠？ | 帶餐者 |
| 登入與發單流程是否太繁瑣？ | 全體 |

## 9. Sprint 2：媒合流程

### Sprint Goal

讓帶餐者能接下訂單，並讓訂餐者知道訂單已被接單與購買中。

### Sprint 2 Story

| ID | Story | Priority | Activity | 說明 |
|---|---|---|---|---|
| AG-004 | 接單 | P0 | A4 接單 | 帶餐者可接下待接訂單 |
| AG-005-1 | 更新訂單狀態：已接單 / 購買中 | P0 | A5 狀態更新 | 帶餐者可將訂單更新為購買中 |
| AG-003-1 | 訂單列表狀態調整 | P1 | A3 瀏覽訂單 | 已接單訂單不應繼續出現在待接列表 |

### Sprint 2 Demo Flow

1. 訂餐者建立訂單
2. 帶餐者查看待接訂單
3. 帶餐者點擊接單
4. 訂單狀態變為「已接單」
5. 帶餐者點擊「開始購買」
6. 訂單狀態變為「購買中」
7. 訂餐者可看到狀態變化

### Sprint 2 驗收重點

| Story | 驗收重點 |
|---|---|
| AG-004 | 接單後訂單狀態變為已接單 |
| AG-004 | 已被接走的訂單不可被其他人再次接單 |
| AG-005-1 | 只有接單者可以更新狀態 |
| AG-003-1 | 已接單訂單不再出現在其他帶餐者的待接列表 |

### Sprint 2 Review 可收集的回饋

| 問題 | 對象 |
|---|---|
| 接單前是否需要更多確認資訊？ | 帶餐者 |
| 狀態「已接單」「購買中」是否容易理解？ | 訂餐者 / 帶餐者 |
| 接單後訂餐者是否能清楚知道有人負責？ | 訂餐者 |
| 是否需要防止帶餐者接自己的訂單？ | 團隊 / 測試者 |

## 10. Sprint 3：交易閉環

### Sprint Goal

讓訂單能完成從購買、送達、確認收餐到評價的完整閉環。

### Sprint 3 Story

| ID | Story | Priority | Activity | 說明 |
|---|---|---|---|---|
| AG-005-2 | 更新訂單狀態：已送達 | P0 | A5 狀態更新 | 帶餐者可標記餐點已送達 |
| AG-006 | 確認收餐 | P0 | A6 完成交易 | 訂餐者可確認收到餐點 |
| AG-007 | 雙向評價 | P1 | A7 建立信任 | 雙方可在完成交易後評分 |

### Sprint 3 Demo Flow

1. 訂餐者建立訂單
2. 帶餐者接單
3. 帶餐者更新為「購買中」
4. 帶餐者更新為「已送達」
5. 訂餐者點擊「確認收餐」
6. 訂單狀態變為「已完成」
7. 雙方留下 1–5 星評價

### Sprint 3 驗收重點

| Story | 驗收重點 |
|---|---|
| AG-005-2 | 帶餐者可將購買中訂單更新為已送達 |
| AG-006 | 只有訂餐者可以確認收餐 |
| AG-006 | 只有已送達狀態才能確認收餐 |
| AG-007 | 只有已完成訂單可以評價 |
| AG-007 | 評價送出後不可修改 |

### Sprint 3 Review 可收集的回饋

| 問題 | 對象 |
|---|---|
| 使用者是否知道什麼時候交易完成？ | 訂餐者 / 帶餐者 |
| 「確認收餐」按鈕是否足夠明顯？ | 訂餐者 |
| 星等評價是否足以建立基本信任？ | 全體 |
| 是否需要文字評論？ | 訂餐者 / 帶餐者 |

## 11. Sprint 4：修正、強化與展示準備

### Sprint Goal

強化 MVP 穩定性，處理基本例外流程，完成期末展示準備。

### Sprint 4 Story

| ID | Story | Priority | Activity | 說明 |
|---|---|---|---|---|
| AG-008 | 取消訂單 | P1 | A2 / A4 例外流程 | 訂餐者可取消尚未被接的訂單 |
| AG-009 | Email 或站內通知 | P2 | A5 狀態更新 | 狀態變更時提醒使用者 |
| AG-010 | 歷史訂單紀錄 | P2 | A6 完成交易 | 使用者可查看過去訂單 |
| AG-011 | UI / Bug Fix | P0 | 全流程 | 修正 Sprint Review 發現問題 |
| AG-012 | Demo Seed Data | P0 | 展示準備 | 建立展示用測試帳號與訂單資料 |

### Sprint 4 Demo Flow

1. 使用測試帳號登入
2. 完整跑一筆交易流程
3. 展示取消待接訂單
4. 展示完成訂單與評價
5. 展示 Review / Retro 後修正的內容

### Sprint 4 驗收重點

| Story | 驗收重點 |
|---|---|
| AG-008 | 待接單可取消，已接單不可取消 |
| AG-009 | 狀態變更時能通知或顯示提醒 |
| AG-010 | 使用者能看到自己的歷史訂單 |
| AG-011 | 主要 Demo flow 不出現阻斷性錯誤 |
| AG-012 | 期末展示資料可穩定重現 |

### Sprint 4 Review 可收集的回饋

| 問題 | 對象 |
|---|---|
| Demo 流程是否清楚？ | 老師 / 同學 |
| 使用者是否能理解平台價值？ | 老師 / 同學 |
| 哪些功能若有下一版最該優先做？ | 全體 |
| MVP 是否已證明核心流程可行？ | 團隊 |

## 14. Not Now

| 多筆順路接單 | 會提高狀態與交易複雜度 |
| 多校區支援 | 期末先聚焦單一校園 |

## 15. Story Map 結論

CampusEats 的 MVP 應優先完成以下交易閉環：

> 登入 → 發布訂單 → 瀏覽訂單 → 接單 → 更新狀態 → 確認收餐 → 評價

Sprint 1 先建立資料流與發單能力。  
Sprint 2 建立媒合能力。  
Sprint 3 完成交易閉環。  
Sprint 4 處理例外、修正與展示準備。

若開發時間不足，應優先保留 AG-001 至 AG-006，確保至少能完成「發單 → 接單 → 狀態更新 → 確認收餐」的核心流程。AG-007 評價建議盡量完成，因為它直接對應本專案的信任機制。
