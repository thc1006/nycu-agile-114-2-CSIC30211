# CampusEats 校園美食媒合平台 User Journey Map

## 1. 文件目的

本文件用來描述 CampusEats MVP 中主要使用者的使用流程、接觸點、痛點、感受與產品機會。

本階段聚焦於兩類核心使用者：

1. 訂餐者：需要他人協助購買餐點的學生
2. 帶餐者：本來就要外出用餐，願意順便幫其他學生帶餐的學生

User Journey Map 將用來協助後續 Story Mapping、Product Backlog 排序、Acceptance Criteria 撰寫與 Sprint Planning。

---

## 2. MVP 使用情境

CampusEats 的 MVP 聚焦於以下端到端流程：

> 訂餐者登入 → 發布訂單 → 帶餐者瀏覽訂單 → 帶餐者接單 → 帶餐者更新狀態 → 訂餐者確認收餐 → 雙方評價

本階段不處理平台金流、GPS 即時追蹤、多校區支援、商家 API 串接與完整管理後台。

## 10. MVP 設計原則

根據以上 Journey Map，本專案 MVP 應遵守以下設計原則：

1. 優先完成端到端交易閉環，而不是先做大量附加功能。
2. 訂單資訊必須清楚，避免帶餐者因資訊不足而不敢接單。
3. 訂單狀態必須簡單明確，避免使用者不知道下一步要做什麼。
4. 評價流程必須輕量，降低使用者完成交易後的操作負擔。
5. 先用手動狀態更新取代 GPS，降低開發成本與隱私風險。
6. 先用星等評價取代完整評論與申訴機制，降低 MVP 複雜度。

## 11. 從 Journey Map 得出的功能優先順序

| 優先級 | 功能 | 原因 |
|---|---|---|
| P0 | 發布訂單 | 沒有訂單就無法形成媒合 |
| P0 | 待接訂單列表 | 帶餐者需要知道有哪些需求 |
| P0 | 接單 | 形成媒合的關鍵行為 |
| P0 | 訂單狀態更新 | 降低等待焦慮，讓流程可追蹤 |
| P0 | 確認收餐 | 明確定義交易完成 |
| P1 | 雙向評價 | 建立基本信任，但可在核心流程後完成 |
| P1 | 取消訂單 | 處理基本例外狀況 |
| P2 | 通知 | 有助於體驗，但可先用頁面狀態替代 |
| P2 | 歷史訂單 | 有助於使用紀錄，但不是第一輪核心流程 |

## 12. 需要在 Review 中驗證的問題

後續 Sprint Review 可根據以下問題收集回饋：

### 訂餐者

1. 發布訂單時，欄位是否足夠清楚？
2. 訂餐者是否知道訂單目前在哪個狀態？
3. 訂餐者是否知道什麼時候該確認收餐？
4. 評價流程是否簡單？

### 帶餐者

1. 待接訂單列表是否足以判斷要不要接單？
2. 接單前是否能看懂餐廳、地點、時間與報酬？
3. 狀態更新按鈕是否清楚？
4. 帶餐者是否知道完成交付後流程結束在哪裡？

---

## 13. Journey Map 結論

從使用者流程來看，CampusEats MVP 應優先處理的不是完整平台功能，而是讓訂餐者與帶餐者能順利完成一筆基本交易。

因此，後續 Story Map 與 Backlog 應以以下流程為主軸：

> 登入 → 發布訂單 → 瀏覽訂單 → 接單 → 狀態更新 → 確認收餐 → 評價

如果 Sprint 時間不足，應優先保留此核心流程，延後通知、歷史訂單、管理後台、文字評論與進階篩選等功能。
