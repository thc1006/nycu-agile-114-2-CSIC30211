# CampusEats 校園美食媒合平台 Project Brief

## 1. 專案名稱

CampusEats 校園美食媒合平台

## 2. 專案背景

在校園生活中，學生常因天氣不佳、課業繁忙、實驗課時間緊湊，或用餐尖峰時段排隊過長，而不方便親自外出購買餐點。

同時，也有另一群學生本來就會外出用餐，且願意順便幫其他同學帶餐，以換取小額報酬或互助回饋。

目前校園內雖然存在「需要幫忙買餐」與「願意順路帶餐」兩種需求，但缺乏一個有效、清楚、可信任的媒合機制。現有方式如 Line 群組或私訊揪團，容易出現資訊分散、訂單混亂、無法追蹤狀態、缺乏評價紀錄等問題。

因此，本專案提出一個校園版餐點媒合平台，讓學生能透過系統發布帶餐需求，也讓願意帶餐的學生能瀏覽並接單，形成「學生幫學生」的共享互助機制。

---

## 3. 產品願景

建立一個專為校園場景設計的點對點餐點媒合平台，讓訂餐者能節省外出與排隊時間，讓帶餐者能在原本外出用餐的路線中取得小額報酬，並透過狀態追蹤與評價機制建立基本信任。

本平台不是職業外送平台，而是校園內學生互助型的帶餐媒合工具。

---

## 4. 目標使用者

### 4.1 訂餐者

需要他人協助帶餐的學生。

常見情境包含：

- 天氣不好，不想外出買飯
- 課業或實驗繁忙，無法離開教室或實驗室
- 用餐時間有限，不想花時間排隊
- 希望能在指定地點取得餐點

主要需求：

- 能快速發布餐點需求
- 能清楚填寫餐廳、餐點、取餐地點與期望時間
- 能知道是否有人接單
- 能追蹤目前訂單狀態
- 能在收到餐點後完成確認與評價

### 4.2 帶餐者

本來就要外出用餐，願意順便幫其他學生帶餐的學生。

常見情境包含：

- 固定會去某間餐廳或校園餐廳
- 願意順路幫忙帶餐
- 希望獲得小額報酬
- 希望接的訂單不要過度影響自身行程

主要需求：

- 能查看目前有哪些待接訂單
- 能根據餐廳、地點、時間、報酬判斷是否接單
- 能接下自己可以完成的訂單
- 能更新購買與送達狀態
- 能在交易完成後評價訂餐者

---

## 5. 核心問題

本專案要處理的核心問題是：

> 校園中存在帶餐需求與順路帶餐供給，但目前缺乏一個能清楚發布需求、媒合帶餐者、追蹤訂單狀態並建立基本信任的流程。

因此，本專案不優先解決完整金流、GPS 即時定位、餐廳系統串接或跨校區營運，而是先聚焦在「校園內學生之間的帶餐媒合流程是否可行」。

---

## 6. MVP 目標

本期末專案的 MVP 目標是：

> 在期末展示前，完成一個可操作的校園帶餐媒合流程，讓使用者能完成「登入 → 發布訂單 → 瀏覽訂單 → 接單 → 更新狀態 → 確認收餐 → 評價」的完整閉環。

MVP 不追求完整商業化，而是驗證以下問題：

1. 訂餐者是否能清楚發布帶餐需求？
2. 帶餐者是否能理解並接下適合的訂單？
3. 雙方是否能透過狀態更新掌握交易進度？
4. 完成交易後是否能透過評價建立基本信任？
5. 此流程是否足以作為後續擴充的基礎？

---

## 7. MVP 功能範圍

### 7.1 In Scope

本次期末專案會優先完成以下功能：

| 功能模組 | 說明 |
|---|---|
| 帳號登入 | 使用者可註冊、登入、登出 |
| 發布訂單 | 訂餐者可建立餐點需求訂單 |
| 待接訂單列表 | 帶餐者可查看目前可接的訂單 |
| 接單 | 帶餐者可接下某一筆待接訂單 |
| 訂單狀態更新 | 帶餐者可更新訂單狀態，例如已接單、購買中、已送達 |
| 確認收餐 | 訂餐者可確認收到餐點並完成交易 |
| 雙向評價 | 訂餐者與帶餐者可在完成交易後給予星等評分 |
| 基本訂單紀錄 | 使用者可查看自己建立或接下的訂單 |

---

## 8. Out of Scope

以下功能不列入本次 MVP，若時間允許可列為 Future Work：

| 排除功能 | 排除原因 |
|---|---|
| 平台內建金流 | 金流與法規複雜度高，超出期末專案主要範圍 |
| 即時 GPS 追蹤 | 涉及隱私與技術複雜度，MVP 先以手動狀態更新替代 |
| AI 餐廳推薦 | 非核心媒合流程，先不實作 |
| 多校區支援 | 先聚焦單一校園場景 |
| 商家 API 串接 | 需要外部商家合作，超出學生專案範圍 |
| 完整申訴與管理後台 | 可作為後續擴充，不作為核心 MVP |
| 多人共同訂餐或拆單 | 流程複雜度高，留待後續版本 |

---

## 9. 成功標準

由於本專案是期末專案，不採用 6 個月 MAU、留存率或真實營運指標作為主要評估標準。期末階段成功標準如下：

| 指標 | 成功標準 |
|---|---|
| 核心流程完成度 | Demo 時可完整跑完一筆交易流程 |
| Story 完成度 | Sprint 內選定的 P0 stories 可操作且符合 AC |
| 使用者理解度 | 測試者能理解如何發單、接單與完成交易 |
| 狀態可追蹤性 | 訂餐者能看見訂單狀態變化 |
| 信任機制 | 完成交易後可進行基本星等評價 |
| 敏捷流程證據 | 具備 Sprint Planning、Sprint Review、Sprint Retrospective 紀錄與影片 |

---

## 10. 初步 Sprint 規劃

本專案至少規劃 4 個 Sprint，以符合期末專案需要足夠進行多次迭代的要求。

| Sprint | Sprint Goal | 預期成果 |
|---|---|---|
| Sprint 0 | 完成產品探索與開發準備 | Project Brief、Impact Map、Story Map、Backlog、初步 UI / API 規劃 |
| Sprint 1 | 建立帳號與發單基礎流程 | 使用者可登入並建立訂單 |
| Sprint 2 | 完成瀏覽訂單與接單流程 | 帶餐者可查看待接訂單並接單 |
| Sprint 3 | 完成狀態更新與收餐確認 | 訂單可從已接單進展到已完成 |
| Sprint 4 | 完成評價、修正與展示準備 | 雙方可評價，系統可進行完整 Demo |

---

## 11. 主要假設

本專案目前基於以下假設：

1. 學生願意透過平台發布帶餐需求。
2. 部分學生願意在原本外出用餐時順便幫忙帶餐。
3. 小額報酬或互助回饋足以吸引帶餐者接單。
4. 對於 MVP 來說，手動狀態更新足以取代即時定位。
5. 基本星等評價能初步建立雙方信任。
6. 先聚焦單一校園，比一開始支援多校區更適合期末專案。

---

## 12. 主要風險

| 風險 | 說明 | 初步應對 |
|---|---|---|
| 使用者不願意接單 | 帶餐者可能覺得報酬不足或流程麻煩 | 在 Story Map 中優先設計清楚的訂單資訊與報酬顯示 |
| 訂單資訊不清楚 | 帶餐者可能買錯餐點或送錯地點 | 發布訂單表單需設計必填欄位與確認摘要 |
| 狀態未即時更新 | 訂餐者可能不知道目前進度 | 將狀態更新列為 MVP 核心功能 |
| 評價機制不足 | 只靠星等可能無法完整反映信任 | MVP 先做星等，文字評論列為後續擴充 |
| 範圍過大 | 功能太多可能導致 Sprint 做不完 | 先完成 P0 核心流程，其他功能列為 Future Work |

---

## 13. PM / Scrum Master 初期工作

PM / Scrum Master 在專案初期需要先完成以下工作：

1. 整理並維護 Project Brief
2. 收斂 MVP Impact Map
3. 建立 User Journey Map
4. 建立 Story Map 並切分 Sprint
5. 整理 Product Backlog 與優先順序
6. 至少撰寫 5 個核心 Story 的 Acceptance Criteria
7. 規劃 Sprint Planning、Sprint Review、Sprint Retrospective 的紀錄方式與影片安排
